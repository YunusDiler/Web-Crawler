# Recommendations for Production Deployment

The most impactful step toward production readiness is replacing SQLite with a proper distributed data layer. PostgreSQL (with its robust concurrent-write support) or a combination of Redis for the URL frontier queue and Elasticsearch for the search index would allow the system to scale horizontally. The current single-process thread pool should be replaced with a task queue like Celery or a custom worker pool behind a message broker (RabbitMQ or Redis Streams), enabling multiple crawler workers across machines. Each worker would pull URL batches from the shared queue, apply per-domain rate limiting (a politeness policy keyed by hostname, not just a global rate), and write results back to the shared index. DNS resolution should be cached aggressively, and a headless browser pool (Playwright or Puppeteer) should be available for JavaScript-rendered pages. The back-pressure system should evolve from simple queue-depth watermarks into adaptive feedback that accounts for target-host response latency, error rates, and HTTP 429/retry-after signals.

On the operational side, the system needs structured logging shipped to a centralized platform (ELK or Datadog), Prometheus metrics for queue depth, crawl throughput, error rates, and per-domain latency, and alerting on back-pressure state transitions. The search component should move to Elasticsearch or Typesense for proper BM25 ranking, stemming, and phrase matching — the current term-frequency model is a reasonable placeholder but lacks the sophistication users expect. For durability, the URL frontier should use write-ahead journaling with checkpointing so that a crash mid-batch loses at most a few seconds of work. Finally, the system should respect `robots.txt` `Crawl-delay` directives, implement per-domain politeness delays, and support configurable crawl policies (same-domain-only, URL pattern whitelists, content-type filters) to ensure the crawler is a good citizen of the web at scale.
